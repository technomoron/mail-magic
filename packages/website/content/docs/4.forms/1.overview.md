# Form Overview

Mail Magic supports public form submissions as a first-class use case. This page covers the public endpoint contract,
the `form_key` concept, and recipient routing.

## Public Form Endpoint Contract

Endpoint:

```text
POST /api/v1/form/message  (no auth required)
```

### Required fields

| Field | Type | Description |
|---|---|---|
| `_mm_form_key` | string | The form key identifying which form template to use |

### Optional system fields

| Field | Type | Description |
|---|---|---|
| `_mm_locale` | string | Locale hint (exposed to the template, does not change form selection) |
| `_mm_recipients` | string[] or comma-separated string | Recipient idnames to route the message to |

### CAPTCHA token fields

Accepted as-is in the provider's native field name:

| Field | Provider |
|---|---|
| `cf-turnstile-response` | Cloudflare Turnstile |
| `h-captcha-response` | hCaptcha |
| `g-recaptcha-response` | Google reCAPTCHA |
| `captcha` | Generic / legacy |

### File attachments

Attachment field names must start with `_mm_file` (e.g. `_mm_file1`, `_mm_file2`). The server enforces
`FORM_MAX_ATTACHMENTS` and `UPLOAD_MAX` per file.

### User fields

All other non-`_mm_*` fields are accepted as user fields and are exposed to the template as `_fields_`. If the form
template has `allowed_fields` configured, `_fields_` is filtered to that allowlist (plus `email`, `name`,
`first_name`, `last_name` which are always allowed).

## The `form_key` Concept

`POST /api/v1/form/template` returns a stable, randomly-generated `form_key`. Public submissions use that key as
`_mm_form_key` to identify which form to process.

Treat `form_key` as sensitive: anyone who has it can submit to that form. Use long, random keys (Mail Magic generates
them automatically), and rotate them if you suspect a leak.

Ignored legacy fields (treated as user fields, not used for routing or authentication):

- `domain`, `formid`, `secret`, `recipient`, `recipient_idname`, `replyto` (and casing variants)

## Recipient Allowlist and Reply-To

Mail Magic supports a recipient allowlist stored server-side. This allows public forms to have a "send to: Sales /
Support" selector without exposing real email addresses in the browser.

### Setting up recipients (authenticated)

```bash
curl -X POST http://localhost:3776/api/v1/form/recipient \
  -H "Authorization: Bearer apikey-<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "example.test",
    "idname": "support",
    "email": "Support Team <support@example.test>"
  }'
```

Each recipient mapping has an `idname` and an email address. Mappings can be:

- **Form-scoped:** provide `form_key` when upserting the mapping — applies only to that form
- **Domain-wide fallback:** omit `form_key` — applies to all forms on the domain

Form-scoped mappings override domain-wide mappings for the same `idname`.

### Submitting with recipient routing (public)

```bash
curl -X POST http://localhost:3776/api/v1/form/message \
  -H "Content-Type: application/json" \
  -d '{
    "_mm_form_key": "<form_key>",
    "_mm_recipients": ["support"],
    "name": "Kai",
    "email": "kai@example.test"
  }'
```

If `_mm_recipients` is omitted, the form's stored default recipient is used.

### Reply-To configuration

Reply-To behavior is configured per form (stored with the form template):

| Field | Description |
|---|---|
| `replyto_from_fields` | When `true`, derive Reply-To from submitted `email` + `name`/`first_name`/`last_name` |
| `replyto_email` | Forced reply-to mailbox, or fallback when extraction fails |
| `allowed_fields` | Optional allowlist of field names exposed to templates as `_fields_` |

Precedence:

- If `replyto_from_fields=true`: use extracted Reply-To if possible, otherwise fall back to `replyto_email` (if set).
- If `replyto_from_fields=false`: use `replyto_email` (if set).
- Otherwise: omit Reply-To.
