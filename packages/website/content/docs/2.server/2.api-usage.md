# API Usage

## Authentication

All authenticated routes require a Bearer token in the `Authorization` header:

```text
Authorization: Bearer apikey-<user_token>
```

Tokens are stored as `HMAC-SHA256(token, API_TOKEN_PEPPER)` in the database. You can seed a plaintext `token` in
`init-data.json`; it will be HMACed on import and the plaintext cleared.

## Transactional Templates

### Store / update a template (authenticated)

```bash
curl -X POST http://localhost:3776/api/v1/tx/template \
  -H "Authorization: Bearer apikey-<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "example.test",
    "name": "welcome",
    "sender": "Example <noreply@example.test>",
    "subject": "Welcome",
    "locale": "en",
    "template": "<p>Hi {{ _vars_.first_name }}</p>"
  }'
```

### Send a transactional message (authenticated)

```bash
curl -X POST http://localhost:3776/api/v1/tx/message \
  -H "Authorization: Bearer apikey-<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "example.test",
    "name": "welcome",
    "locale": "en",
    "rcpt": "person@example.test",
    "vars": { "first_name": "Ada" }
  }'
```

You can send to multiple recipients by providing `rcpt` as an array:

```json
{
  "domain": "example.test",
  "name": "welcome",
  "rcpt": ["ada@example.test", "kai@example.test"],
  "vars": { "first_name": "Ada" }
}
```

## Form Templates

### Store / update a form template (authenticated)

This returns `data.form_key`, which is used by the public submission endpoint.

```bash
curl -X POST http://localhost:3776/api/v1/form/template \
  -H "Authorization: Bearer apikey-<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "example.test",
    "idname": "contact",
    "sender": "Example Forms <forms@example.test>",
    "recipient": "owner@example.test",
    "subject": "New contact form submission",
    "locale": "en",
    "template": "<p>Contact from {{ _fields_.name }} ({{ _fields_.email }})</p>"
  }'
```

### Submit a form (public endpoint, no auth)

```bash
curl -X POST http://localhost:3776/api/v1/form/message \
  -H "Content-Type: application/json" \
  -d '{
    "_mm_form_key": "<form_key from the template response>",
    "name": "Kai",
    "email": "kai@example.test",
    "message": "Hello"
  }'
```

### Submit a form with file attachments (multipart)

Attachment field names must start with `_mm_file`:

```bash
curl -X POST http://localhost:3776/api/v1/form/message \
  -F "_mm_form_key=<form_key>" \
  -F "name=Kai" \
  -F "email=kai@example.test" \
  -F "_mm_file1=@brief.pdf"
```

## Recipient Allowlist

Configure named recipients server-side so public forms can route without exposing real email addresses.

### Add a recipient mapping (authenticated)

```bash
curl -X POST http://localhost:3776/api/v1/form/recipient \
  -H "Authorization: Bearer apikey-<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "domain": "example.test",
    "idname": "support",
    "email": "Support Team <support@example.test>"
  }'
```

### Submit with recipient routing (public)

```bash
curl -X POST http://localhost:3776/api/v1/form/message \
  -H "Content-Type: application/json" \
  -d '{
    "_mm_form_key": "<form_key>",
    "_mm_recipients": ["support"],
    "name": "Kai",
    "email": "kai@example.test",
    "message": "Hello"
  }'
```

## Assets

### Upload a domain asset (authenticated)

```bash
curl -X POST http://localhost:3776/api/v1/assets \
  -H "Authorization: Bearer apikey-<token>" \
  -F "domain=example.test" \
  -F "file=@logo.png"
```

## Swagger / OpenAPI

When `SWAGGER_ENABLED=true` is set, the full OpenAPI spec is served at:

```text
GET /api/swagger
```

Feed this JSON to Swagger UI, Postman, or Insomnia to explore and test all endpoints interactively. The spec is the
canonical reference for exact request/response shapes, status codes, and error bodies.

The packaged spec is also available on disk at:

```text
node_modules/@technomoron/mail-magic/docs/swagger/openapi.json
```
